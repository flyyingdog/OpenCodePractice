# 旧 Java 仓补 Spec 操作说明

## 目标

在不阻塞当前开发的前提下，为已有 Java 系统逐步建立 OpenSpec 能力地图与当前行为说明。

## 适用场景

- 代码仓之前没有使用过 spec workflow
- 当前无法快速判断新需求应该挂到哪个已有功能
- 团队希望建立长期可维护的需求与设计知识库

## 核心原则

1. 不做一次性全量补齐。
2. 只补当前活跃改动会触达的能力。
3. 先补“最小现状 spec”，再逐步增强。
4. spec 按业务能力组织，不按技术层组织。

## 推荐步骤

### 1. 先建立能力索引

根据业务域和对外能力，先列出系统最主要的能力，例如：

- 用户认证
- 用户资料
- 订单生命周期
- 支付结算
- 库存扣减
- 消息通知

每个能力在 `openspec/specs/<capability>/spec.md` 下保留一个长期 spec。

### 2. 为能力写最小现状 spec

每个现状 spec 至少包含：

- 能力目的
- 3 到 8 条核心 Requirement
- 每条 Requirement 下面 1 到 3 个关键 Scenario

只写当前系统已经存在且稳定的业务行为，不写代码实现细节。

### 3. 新需求到来时做挂靠判断

对每个新变更，先判断它属于哪一类：

- 修改已有能力：在对应 capability 上增加或调整 delta
- 新增独立能力：创建新的 capability spec
- 横跨多个能力：允许一个 change 影响多个 spec

### 4. 找不到归属时的处理办法

如果一开始无法确定需求对应哪个能力：

1. 先根据业务目标选一个最可能的 capability
2. 在 proposal 和 design 中明确标注“当前为临时挂靠”
3. 在任务里加入“整理或拆分相关长期 spec”的后续任务

## Java 项目补充建议

### 读代码时优先寻找这些线索

- `controller` / `resource`：外部接口边界
- `service` / `application`：用例和业务编排
- `domain`：实体、聚合和值对象
- `repository` / `dao` / `mapper`：持久化边界
- `entity` / `po` / `do`：数据模型
- `scheduler` / `listener` / `consumer`：异步和后台处理

### 一个能力是否应该单独建 spec

可以用这几个判断标准：

- 是否有独立业务目标
- 是否有稳定对外行为
- 是否有明确边界和规则
- 是否会被多个需求反复修改

如果这几个问题大多回答“是”，就值得单独建 spec。

## 常见误区

- 误区：先补所有历史功能再开发
- 建议：先让当前开发跑起来，按改动路径渐进回填

- 误区：spec 直接按代码目录组织
- 建议：spec 应该优先映射业务能力

- 误区：把每次 change 的 design 直接拼成一个总设计文档
- 建议：保留 change 设计历史，同时维护单独的长期设计文档
