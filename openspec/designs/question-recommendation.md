# 问题推荐与问题联想设计

## 背景

问题推荐与问题联想模块用于根据当前上下文和历史数据生成候选问题，帮助用户延续对话、发现关联主题，并提升整体交互效率。

当前已知存在多条推荐路径：

- 基于历史记忆推荐
- 基于预置典型问题库提取
- 基于历史记忆中的实体提取进行推荐

这些路径可以拥有不同的请求参数、处理逻辑和结果模型，但需要在系统层面维持统一的能力边界。

## 目标

- 支持多策略问题推荐
- 支持不同策略的独立接口和模型演进
- 支持统一的结果聚合与来源标识
- 支持随着记忆系统演进不断扩展新策略

## 非目标

- 不负责历史记忆原始存储
- 不负责长期记忆学习过程
- 不限定最终只能使用单一推荐算法

## 模块边界

### 上游

- 当前会话上下文
- 工作记忆与历史记忆
- 长期记忆
- 预置问题库
- 实体抽取服务或实体识别逻辑

### 下游

- 前端候选问题展示
- 对话引导与交互增强
- 运营推荐分析

## 推荐策略模型

### 1. 基于历史记忆推荐

输入：

- 当前会话上下文
- 历史问答摘要
- 可选的长期记忆召回结果

输出：

- 与当前语境连续性最强的候选问题

适用场景：

- 用户处于连续追问过程
- 希望延续当前话题

### 2. 基于预置典型问题库提取

输入：

- 当前上下文特征
- 问题库配置

输出：

- 与当前主题最相关的典型问题

适用场景：

- 需要引导用户探索常见问题
- 需要结合业务预置知识进行推荐

### 3. 基于实体提取的推荐

输入：

- 历史记忆中的实体
- 实体关联规则或模板

输出：

- 围绕关键实体延展出的候选问题

适用场景：

- 对话中出现高价值对象、概念、主题词
- 希望围绕实体开展深入追问

## 统一结果模型

不同策略可以有各自内部模型，但建议对外统一候选问题结果结构。

### 建议字段

| 字段 | 说明 |
| --- | --- |
| `question` | 推荐问题文本 |
| `sourceType` | 推荐来源，如 history / preset / entity |
| `strategyId` | 生成该结果的策略标识 |
| `score` | 推荐分数 |
| `reason` | 推荐原因摘要 |
| `entityRefs` | 关联实体，可选 |
| `metadata` | 扩展信息 |

## 接口设计原则

### 统一聚合接口

适用场景：

- 前端只关心最终候选问题列表
- 系统需要屏蔽内部多策略复杂性

职责：

- 调用多个推荐策略
- 汇总结果
- 去重、裁剪、排序

### 独立策略接口

适用场景：

- 单独调试某一策略
- 对不同策略做独立演进和评估

职责：

- 暴露策略专属参数
- 返回策略原生结果

## 聚合与排序

建议聚合阶段至少考虑：

- 去重
- 相似问题合并
- 来源保留
- 多策略分数归一化
- 业务优先级控制

排序因素可包括：

- 与当前上下文相关性
- 用户近期兴趣
- 预置问题优先级
- 实体重要度

## 兼容性策略

- 不同推荐方式可以独立演进请求参数
- 聚合接口应尽量保持稳定字段
- 当返回模型变更时，应明确版本或兼容策略

## 可观测性

建议记录：

- 每次推荐请求使用了哪些策略
- 每个策略返回了多少候选问题
- 最终被聚合输出的问题及其来源
- 用户点击或采纳反馈

## 演进方向

- 支持更多推荐策略
- 引入反馈闭环优化排序
- 支持按用户群体或业务场景差异化推荐
- 支持多语言或多领域问题联想

## 风险与注意事项

- 多策略结果如果没有统一评分语义，聚合质量会不稳定
- 如果来源标识丢失，后续无法解释为什么推荐该问题
- 预置问题库若缺乏治理，可能输出过时或低质量问题
